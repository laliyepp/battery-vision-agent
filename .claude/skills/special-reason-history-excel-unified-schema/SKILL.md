# Special Reason History Excel — Unified Schema

Read a historical parameter markdown file (converted from Excel) and extract all vehicle/battery/motor/econtrol data **directly into the canonical unified schema structure** defined in `schema/unified_schema_v2.json`.

This is the same logic as `/reason-excel-unified-schema` but with a custom output filename, report_id, and application date. The markdown file is pre-generated by `excel_to_markdown.py` in a prior pipeline step.

## Arguments

- `$0` — Path to the markdown file (`_excel.md`, already converted from Excel)
- `$1` — Path to the output directory
- `$2` — Report ID (used for output filename and 检测报告文件名)
- `$3` — Application date (used for 签发日期), may be empty string

## Output

- `$1/$2_unified_output.json` — Same 4-domain structure as `_unified_output.json` files

## Process

1. **Skip check**: If `$1/$2_unified_output.json` already exists, print `[Skip] $2_unified_output.json already exists` and stop.

2. **Read the schema**: Read `schema/unified_schema_v2.json` from the project root. This defines all 4 domains (通用, 电池, 电机, 电控) with their fields, descriptions, and keywords.

3. **Read the markdown**: Read the file at `$0` using the Read tool. This markdown file was generated by `excel_to_markdown.py` in a prior pipeline step.

4. **Initialize** the output template — a JSON object with all 4 domains, each containing every schema key set to `""`.

5. **Auto-fill 通用 domain**:
   - Fill `车辆名称` and `车辆型号` from the markdown content.
   - Fill `生产单位` from the markdown content if available.
   - Set `检测报告文件名` = `$2` (the report ID, NOT `"declared"`).
   - Set `签发日期` = `$3` (the application date — may be `""`).
   - Leave other report-specific fields as `""`: `受检单位`, `检验单位`, `检验类别`, `检验依据`, `送样日期`, `检验日期`.

6. **Extract 电池/电机/电控 data**: Reason through the markdown content to fill in values. For each schema field:
   - Use the `description` to understand what kind of information belongs there.
   - Use the `keywords` to locate matching parameters in the markdown tables.
   - If the markdown contains a clear, explicit value for the field, fill it in as a string.
   - If no matching data is found, leave the value as `""`.

7. **Write** the completed JSON to `$1/$2_unified_output.json`.

8. **Print**: `[Done] $2_unified_output.json`

## Output Format

Every output file has the **exact same structure** — all 4 domains, all keys present:

```json

    "车辆名称": "纯电动轿车",
    "车辆型号": "TSL7000BEVBA3",
    "检测报告文件名": "参数表附件整车参数_26667398",
    "受检单位": "",
    "生产单位": "特斯拉（上海）有限公司",
    "检验单位": "",
    "检验类别": "",
    "检验依据": "",
    "送样日期": "",
    "签发日期": "2020-08-31",
    "检验日期": ""
```

## Rules

### Structure
- **ALL schema keys** must appear in every output — use `""` for fields with no data found.
- **No extra keys** beyond what the schema defines.
- **No nested objects** — flat key-value pairs within each domain.
- **All values are strings** — even numbers should be written as strings (e.g., `"220"` not `220`).

### Multi-Value Handling
- When the Excel contains multiple components of the same type (e.g., front and rear motors, or multiple charging sockets), use `||` to separate values.
- **Keep the order synced** across all related columns. For dual motors: always put the front motor first, rear motor second. If front/rear is not labeled, use document order.
- Example: `"电机.型号": "3D3||3D6"`, `"峰值功率_kW": "137||220"`, `"电机.生产企业": "特斯拉（上海）有限公司||特斯拉（上海）有限公司"`.
- If only one component exists, do NOT use `||` — just write the single value.

### Domain Classification
- **电机控制器** data belongs in the **电控** domain, NOT 电机, even when it appears alongside motor parameters in the Excel.
- **BMS** data belongs in the **电池** domain.
- **通用** fields (车辆名称, 型号, 生产单位) apply to the vehicle level.

### Key Differences from reason-excel-unified-schema
- Input is a markdown file (`$0`), not an Excel file — the Excel→markdown conversion happens in a prior pipeline step
- Output filename: `$2_unified_output.json` (not `三电参数_unified_output.json`)
- `检测报告文件名` = `$2` (not `"declared"`)
- `签发日期` = `$3` (not empty)

### Data Quality
- **Do not guess or infer** values. Only fill in data that is explicitly stated in the markdown.
- **Preserve units** as they appear. If the schema column already includes a unit suffix (e.g., `_kW`, `_V`), write only the numeric value as a string.
- **Skip** if output file already exists (do not re-process).
